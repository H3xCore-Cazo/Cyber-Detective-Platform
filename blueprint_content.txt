1. Core Philosophy
Cyber Detective is built on a fundamentally different educational model than existing cybersecurity platforms. While other platforms teach attack skills, Cyber Detective builds investigative minds.

1.1 The Paradigm Shift
Traditional Platform
Cyber Detective
Attack a vulnerable machine
Receive a real incident to investigate
Exploit → Capture Flag
Analyze → Trace → Report
Learn offensive tools
Learn investigative thinking
Hacker mindset
Analyst / Detective mindset
Solo exploit
Full incident reconstruction

1.2 The Investigation Flow
Step 1
Incident Happens — a simulated breach occurs inside the lab environment
Step 2
Collect Evidence — user gathers logs, PCAP, memory dumps, emails
Step 3
Analyze Artifacts — user inspects files, applies forensic techniques
Step 4
Trace Attacker — timeline reconstruction, lateral movement mapping
Step 5
Find Initial Access — identify patient zero, entry vector
Step 6
Identify Persistence — find backdoors, scheduled tasks, registry keys
Step 7
Determine Impact — what data was touched, exfiltrated, modified
Step 8
Write Incident Report — structured DFIR report with IOCs and recommendations

1.3 Who The User Becomes

Threat Hunter

1.4 Why This Succeeds
Companies hire people who can analyze incidents and write reports — not just run tools
DFIR is a rapidly growing field with increasing demand and shortage of trained professionals
The story-based format makes learning engaging — more like a detective game than a course
No other major platform focuses exclusively on the defensive / investigative side

Free text:   validated by server-side regex (never shown to user)

7.4 Scoring Algorithm
Base Points
Defined per objective (e.g., 100 XP)
Time Bonus
+20 XP if solved faster than the case average
First Blood
+50 XP for being the very first solver globally
No Hints Bonus
+25 XP if solved without using any hints
Perfect Report
+50 XP for AI-scored report above 90/100
Hint Penalty
-10 XP per hint consumed (stacks)
Lab Expired
Remaining objectives score at 50% if lab timed out

7.5 Anti-Cheat Measures
All flag validation happens server-side only — flag values never sent to client
Attempt tracking — excessive wrong attempts flagged for review
Time-based anomaly detection — unusually fast solves reviewed
IP-based rate limiting on flag submission endpoint
Browser fingerprinting (future) for repeat offender tracking

8. Investigation Report System
The Investigation Report is the feature that most differentiates Cyber Detective from every other platform. It mirrors what a real DFIR analyst submits after an incident.

8.1 Report Sections
Executive Summary — 1-paragraph overview written for non-technical management
Incident Timeline — Chronological list of events with timestamps
Initial Access — How the attacker entered the environment
Affected Systems — List of compromised hostnames, IPs, accounts
Attacker Actions — Detailed TTPs observed during the investigation
Data Impact — What data was accessed, modified, or exfiltrated
Indicators of Compromise (IOCs) — Structured: IPs, domains, file hashes, paths
Root Cause Analysis — Why was the attack possible?
Recommendations — Actionable steps to prevent recurrence

8.2 AI Report Scoring
After submission, an AI model grades the report across 5 dimensions:
Completeness
Are all sections filled with meaningful content?
Accuracy
Do the findings match what the evidence actually shows?
Professionalism
Grammar, structure, clarity, and appropriate tone
Technical Depth
Specific findings, exact IOCs, precise event IDs
Recommendations Quality
Are suggestions actionable and specific?

Scoring range: 0 – 100. Score breakdown shown with per-section feedback. Score contributes to total XP for the case.

10. Lab Orchestrator — The Brain
The Lab Orchestrator is a dedicated service (separate from the Next.js frontend) responsible for all infrastructure lifecycle management.

10.1 Orchestrator Responsibilities
Create and configure lab sessions on demand
Allocate resources (containers or VMs) based on lab type
Create isolated network per session
Inject evidence files into running environments
Generate and securely store session credentials
Monitor session health via heartbeat
Auto-terminate expired sessions (cron every 60 seconds)
Clean up resources (stop containers, remove networks, delete VMs)
Queue management when resources are at capacity
Track usage metrics for cost analysis and scaling decisions

10.2 Full Lab Start Flow
Step 1
User clicks [Start Investigation] → POST /api/labs/start
Step 2
Validate: user authenticated, case exists, user within concurrent lab limit
Step 3
Create lab_sessions record in database (status: pending)
Step 4
Resource scheduler checks available capacity
Step 5
If capacity available: proceed; if full: add to queue, return position
Step 6
Create isolated Docker network or VLAN for this session
Step 7
Start container(s) or clone/start VM(s)
Step 8
Run evidence injection script inside environment
Step 9
Generate unique username/password for this session
Step 10
Start ttyd or configure Guacamole connection
Step 11
Update database: status = running, store connection_url and credentials
Step 12
Return connection URL to client — lab opens in browser

10.3 Session Lifecycle Management
Heartbeat
Client sends ping every 30s; if missed for 5 min, lab paused
Warning
At T-5 minutes: notification sent via WebSocket to user
Expiration
At T=0: status → expiring, final score saved
Cleanup
Cron job: stop containers/VMs, remove networks, update DB → terminated
Extension
Pro users may extend by 30 or 60 minutes once per session
Pause
Save state if supported (Docker) — VM snapshots for Windows

10.4 Scaling Architecture
Phase 1 — Single VPS (MVP)
1 server: 8 vCPU, 32 GB RAM
Docker Engine directly
Max ~50 concurrent Linux labs
No Windows AD labs yet
Phase 2 — Multi-Server
Kubernetes cluster: 3 worker nodes
Separate VM host (Proxmox) for Windows AD labs
PostgreSQL with read replica
Redis for queuing and session cache
Phase 3 — Cloud Scale
AWS EKS or GKE for container orchestration
Spot instances for cost-efficient Linux labs
Dedicated VM pool for Windows AD (reserved instances)
Auto-scaling based on queue depth
Multi-region for low latency globally

15. Security Principles
15.1 Lab Isolation
Every lab runs in its own isolated Docker network or VLAN
No communication between different users' lab networks
No internet access from lab containers (egress blocked)
Containers run as non-root user (nobody / investigator)
Resource limits enforced: CPU, RAM, disk, network bandwidth
Seccomp profiles applied to restrict dangerous syscalls
AppArmor or SELinux profiles on VM hosts

15.2 Data Protection
All passwords hashed with bcrypt (cost factor 12+)
Flag values stored hashed (SHA-256) — never in plaintext
API keys and secrets stored in environment variables only — never in code
TLS 1.3 enforced for all connections
HSTS preload enabled
Database encryption at rest (Supabase/AWS managed)
Evidence files encrypted in object storage

15.3 Input Validation & Access Control
All user inputs validated with Zod schemas on the server
SQL injection prevention via Prisma parameterized queries
XSS prevention via React (JSX auto-escapes all dynamic values)
CSRF tokens on all state-changing requests
Row Level Security (RLS) in Supabase — users can only read own data
Flag submission: server-side comparison only — flag value never exposed to client

15.4 Rate Limiting
Global: 100 requests per 15-minute window per IP
Flag submission: 10 attempts per minute per session
Lab start: 5 labs per hour per user (free tier)
AI queries: 20 per hour per user (free tier)
Auth endpoints: 5 failed attempts triggers temporary lockout

16. AI Investigator — The Differentiator
The AI Investigator is an embedded assistant inside every active lab session. It is the feature that sets Cyber Detective apart from every other platform — providing contextual guidance without just giving away answers.

16.1 What It Can Do
Explain what a specific Event ID means and why it matters
Explain suspicious log entries in plain language
Suggest what to investigate next based on current findings
Give progressive hints — vague at first, more specific on repeat asks
Validate a user theory: give progressive confirmation or redirection
Teach a concept: "Kerberoasting is when an attacker requests a service ticket..."
Summarize all evidence found so far into a running timeline
Help structure the investigation report

16.2 What It Cannot Do
Reveal flag values directly — ever
Execute commands or interact with the lab environment
Access information outside the current case context
Discuss topics unrelated to the active investigation

16.3 AI Architecture
Model
Claude 3.5 Sonnet or GPT-4o (with fallback)
Context Window
Up to 128k tokens — can include evidence excerpts
System Prompt
DFIR mentor role — progressive hints, never reveal flags, adapt to skill level
Session Context
Includes: case story, evidence accessed, flags already solved, hints given, time remaining
Rate Limit
20 queries/hour free tier, 60/hour Pro
XP Cost
Free initially; potentially 5 XP per query in future (optional)
Safety
Automated tests check for flag leakage, all interactions logged for review

16.4 Hint Level System
Level 1 — Subtle
"Have you examined all the log files? Sometimes the answer is in an unexpected place."
Level 2 — Directive
"Look closely at auth.log around the time of the Nginx alert."
Level 3 — Specific
"Check auth.log line ~847. What's the source IP of the successful SSH login after the failed attempts?"
The AI advances hint levels automatically based on how long the user has been stuck and how many times they have asked.
